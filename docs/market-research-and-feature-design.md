# MemoTask 前期市场调研与功能设计

生成日期：2026-06-21  
阶段：前期产品定义 / 自用 MVP 设计  
目标：做一款手机端和 PC 端数据共通、可随时记录想法，并将想法转化成待办列表的软件。  

## 1. 核心假设

- 主要用户是你本人，因此不设计付费窗口、订阅墙、团队席位、营销落地页或企业管理功能。
- 产品优先解决个人真实工作流：灵感和待办经常出现在不同设备、不同场景里，需要低摩擦记录和可靠同步。
- 记录对象不是纯任务，也不是长文档，而是“未整理的想法、计划、链接、语音、灵感碎片”。
- AI 的角色是“整理助理”：把原始想法拆成 Memo 内部待办、补全上下文；最终排序和取舍权属于用户。
- 首版应优先做响应式 Web / PWA，先覆盖手机和 PC，再考虑原生 App。

## 2. 市场调研摘要

### 2.1 任务管理类

**Todoist**

Todoist 的强项是快速添加任务、自然语言识别日期、提醒、项目、优先级、标签、过滤器、多视图和跨设备同步。它证明了“低摩擦捕获 + 可信待办视图”是任务管理工具的核心护城河。  
参考：https://www.todoist.com/features

对 MemoTask 的启发：

- 快速输入必须是一等功能，不能藏在复杂页面里。
- 快速输入必须比字段填写更重要，用户应该先记录，再整理。
- Todoist 证明任务软件常会走向日期和视图管理；MemoTask 首版要反向收窄，避免 Today / Upcoming 带来的过期压力。

需要避免：

- 不要首版做团队工作区、权限、评论、项目管理模板。
- 不要把个人工具做成项目管理软件。

**TickTick / 滴答清单**

TickTick 覆盖任务、日历、智能筛选、标签、看板、时间线、语音输入、桌面快捷键、手机小组件、重复提醒和日程视图。它的特点是功能非常全，适合重度任务用户。  
参考：https://ticktick.com/features

对 MemoTask 的启发：

- 移动端小组件、桌面快捷入口、语音输入对“随时记录”很关键。
- TickTick 证明入口能力很关键，但 MemoTask 首版只吸收“随时记录”和“拖拽整理”，不吸收日期、日历、重复提醒体系。
- 手机小组件、桌面快捷入口、语音输入可以排到 P1/P2，仍然服务 Memo 捕获，而不是日期计划。

需要避免：

- 不要一次性复制看板、番茄、习惯、日历、艾森豪威尔矩阵。
- 功能太满会稀释“想法转待办”的核心体验。

**Microsoft To Do**

Microsoft To Do 的定位更简单：My Day、列表、提醒、步骤、重要标记、Outlook 集成和跨设备访问。它展示了一个轻量任务工具的基础骨架。  
参考：https://support.microsoft.com/en-au/office/learn-more-about-microsoft-to-do-5bdda953-7494-4cc7-9a46-f0b6c6e3659e

对 MemoTask 的启发：

- “今日计划”是每日打开频率最高的视图。
- 大任务拆成内部小待办是必要功能，但这些待办应留在 Memo 包围框里。
- Memo 详情只保留原文、内部待办、上下文和来源记录，不需要日期、提醒、项目等复杂字段。

### 2.2 轻量记录类

**Google Keep**

Google Keep 提供快速便签、列表、标签、颜色、置顶、归档、提醒和分享。它的优势是打开即写，几乎没有学习成本。  
参考：https://support.google.com/keep/answer/2888240?hl=en

对 MemoTask 的启发：

- 首屏必须能立即输入，而不是先选择分类。
- 标签、置顶、归档比复杂文件夹更适合碎片记录。
- 记录和清单可以共存，但需要明确转化机制。

需要避免：

- Keep 更像便签墙，行动闭环较弱。MemoTask 要补上“记录后如何变成任务”的环节。

**flomo**

flomo 的强项是像聊天一样快速记录、微信输入、多端同步、标签、搜索、每日回顾、相关笔记、API 和 AI 洞察。它明确拒绝 All-in-one，专注记录与回顾。  
参考：https://flomoapp.com/  
参考：https://help.flomoapp.com/

对 MemoTask 的启发：

- “像聊天一样记录”比“打开编辑器写笔记”更适合灵感捕获。
- 回顾机制很重要：很多想法不是马上变成任务，而是在复盘时变成任务。
- API / URL Scheme / 分享入口是个人工作流的放大器。

需要避免：

- flomo 不擅长待办清单，这正是 MemoTask 的机会：记录之后自动进入行动转化。

### 2.3 知识库 / All-in-one 类

**Notion**

Notion 已经从文档和数据库扩展到 AI workspace、项目、知识库、AI Meeting Notes、Agents、Enterprise Search 等方向。它适合组织复杂信息，但对个人随手记录来说启动成本较高。  
参考：https://www.notion.com/product

对 MemoTask 的启发：

- 数据库式结构适合做底层模型，但不适合暴露给首版用户。
- AI 可以做整理、路由、总结，但要避免让用户配置复杂系统。

需要避免：

- 不要把 MemoTask 做成小 Notion。
- 不要在首版提供自定义数据库、页面嵌套、模板市场、协作空间。

**Obsidian**

Obsidian 强调本地文件、离线可用、跨平台、同步、端到端加密、版本历史和可控的数据所有权。  
参考：https://obsidian.md/sync

对 MemoTask 的启发：

- 自用工具应该重视数据归属、导出、离线缓存和恢复。
- 跨端同步必须可解释：什么时候同步、冲突怎么处理、是否有历史版本。

需要避免：

- 不要首版做复杂双链、图谱、插件生态。
- 如果目标是行动管理，知识网络不是核心。

### 2.4 AI 整理类

**Reflect**

Reflect 强调快速同步、网络化笔记、离线移动捕获、端到端加密、日历集成、AI 辅助、语音转录、从散乱想法生成大纲或 action items。  
参考：https://reflect.app/

对 MemoTask 的启发：

- AI 从散乱笔记提炼行动项是明确存在的产品趋势。
- 语音输入和转写是移动端重要入口。
- “AI 生成候选行动，用户确认”比“AI 自动创建一堆任务”更可信。

**Amie**

Amie 近期更聚焦 AI meeting notes 和 action items，强调从会议总结到下一步行动。  
参考：https://amie.so/

对 MemoTask 的启发：

- “上下文 -> 摘要 -> 行动项”是 AI 待办转化的成熟模式。
- MemoTask 可以把这个模式从会议扩展到日常想法、语音、碎片记录。

## 3. 市场空白与产品机会

现有产品大致分为四类：

| 类型 | 代表产品 | 强项 | 对个人想法转待办的缺口 |
| --- | --- | --- | --- |
| 任务管理 | Todoist、TickTick、Microsoft To Do | 待办、提醒、优先级、日程 | 原始想法入口弱，容易要求用户一开始就结构化 |
| 轻量记录 | Google Keep、flomo | 快速记录、标签、回顾 | 行动闭环弱，记录容易沉淀但不执行 |
| 知识库 | Notion、Obsidian | 结构化知识、长期沉淀、可扩展 | 启动成本高，不适合每次灵感捕获 |
| AI 整理 | Reflect、Amie | 摘要、转写、行动项提取 | 常聚焦笔记/会议，不一定服务个人碎片行动流 |

MemoTask 的机会不是成为其中任意一个的替代品，而是做一个更窄的自用工具：

> 用最快方式捕获想法，把每条 Memo 设计成一个大包围框，并让 AI 在框内生成待办项；Memo 的前后顺序就是紧急程度。

这个定位成立的原因：

- 记录类工具让你“记住了”，但不保证你“会做”。
- 待办类工具让你“安排了”，但要求你输入时已经知道任务结构。
- 你要的场景恰好在两者之间：先乱记，再整理成可执行事项。

## 4. 产品定位

### 4.1 一句话定位

MemoTask 是一个跨手机和 PC 的个人想法整理工具：像聊天一样记录 Memo，像助理一样在 Memo 内提取待办，用拖动 Memo 前后顺序来表达紧急程度。

### 4.2 核心价值

- **随时记**：手机、PC、快捷键、分享入口、语音都能快速捕获。
- **自动理**：AI 从原始文本中识别可执行事项，并保留在 Memo 包围框内部。
- **你整理**：用户编辑、勾选、删除 Memo 内部待办，但不需要把它们转成外部任务。
- **拖动排优先级**：Memo 的前后顺序就是紧急程度，越靠前越优先处理。
- **少压力**：首版不做日期、Today、Upcoming、截止提醒和过期任务；后续日期板块只作为用户主动设置的月份地图，避免未完成带来的倦怠感。

### 4.3 非目标

首版不做：

- 付费窗口、会员、订阅、额度墙。
- 团队协作、任务分配、评论、权限。
- 复杂项目管理、甘特图、多人看板。
- 模板市场、公开分享、社区。
- 通用知识库、网页发布、复杂双链图谱。
- 第一版日期计划系统、Today/Upcoming、过期任务、重度日历系统、番茄钟、习惯打卡。

## 5. 推荐方案

### 方案 A：最小可用 PWA

用一个响应式 Web App 同时覆盖手机和 PC，支持安装到桌面和手机主屏。后端提供账号、同步、AI 解析和任务数据。

优点：

- 开发最快，一套代码覆盖多端。
- 适合自用 MVP。
- 后续可平滑演进到移动端壳或桌面壳。

缺点：

- 系统级快捷入口、通知、分享菜单能力不如原生。
- iOS 后台和推送体验可能受限制。

适合阶段：MVP。

### 方案 B：Web + 原生壳

核心仍是 Web，但 PC 端用 Tauri/Electron，移动端用 Capacitor/React Native 包一层原生能力。

优点：

- 快捷键、系统托盘、分享、通知、语音入口更好。
- 用户体验更接近正式 App。

缺点：

- 维护成本更高。
- 首版容易被平台适配拖慢。

适合阶段：P1，当 PWA 的入口能力成为瓶颈后再做。

### 方案 C：本地优先 + 同步引擎

所有记录先写本地数据库，再异步同步到云端；支持离线、冲突合并、历史版本。

优点：

- 自用体验最可靠。
- 手机和 PC 即使网络不稳定也能记录。
- 数据归属和可恢复性强。

缺点：

- 实现复杂度最高。
- 同步冲突、迁移、版本历史需要认真设计。

适合阶段：P1/P2。MVP 可以先做“本地缓存 + 云端权威”，之后演进为完整 local-first。

### 推荐

推荐从 **方案 A：最小可用 PWA** 开始，但在数据模型里预留本地优先和同步版本字段。

原因：

- 你的目标是自用，不需要一开始追求商业级平台覆盖。
- 最关键风险不是技术，而是工作流是否真的每天用。
- PWA 可以最快验证“记录 Memo -> AI 生成内部待办 -> 拖动 Memo 排序”的核心闭环。

## 6. 核心工作流设计

### 6.1 捕获想法

入口：

- 手机首页大输入框。
- PC 首页大输入框。
- PC 全局快捷键，P1。
- 手机分享到 MemoTask，P1。
- 语音输入，P1。
- 浏览器扩展或书签脚本，P2。

输入支持：

- 文本。
- 多行文本。
- 链接。
- 图片附件，P1。
- 语音转文字，P1。

输入后默认成为一张 `Memo` 卡片。Memo 是一个大的包围框，后续所有 AI 生成的待办都保留在这个包围框内部。

示例：

> 研究一下 PWA 能不能覆盖手机和 PC，然后整理 MemoTask 的实现方案。顺便看看有没有更适合拖拽排序的交互库。

系统应生成：

- 一个 Memo 包围框，保留完整原文。
- Memo 内部待办：
  - 调研 PWA 对手机和 PC 的支持情况。
  - 整理 MemoTask 的实现方案。
  - 查找适合拖拽排序的交互库。

### 6.2 AI 解析 Memo 内部待办

AI 解析结果进入 `MemoTodo`，不创建外部独立任务。

AI 识别字段：

- 待办标题。
- 相关上下文。
- 是否可拆步骤。
- 置信度。
- 原始 Memo 引用。

解析策略：

- 短文本：自动解析。
- 长文本：先摘要，再提取内部待办。
- 不提取日期，不建议截止时间，不生成提醒。
- 低置信度待办默认折叠，避免污染 Memo。

### 6.3 Memo 内部整理

AI 生成的待办直接显示在 Memo 包围框内部。

用户可以：

- 勾选完成。
- 编辑待办标题。
- 删除不需要的待办。
- 手动新增待办。
- 调整 Memo 内部待办顺序。
- 标记“只是笔记，不需要待办”。

整理后 Memo 状态变为：

- `active`：仍有未完成待办。
- `done`：内部待办全部完成。
- `note_only`：只是记录。
- `archived`：已归档。

### 6.4 拖动 Memo 排优先级

Memo 本身是主要操作单位。用户通过拖动整个 Memo 包围框的前后顺序来表达紧急程度：

- 越靠前：越紧急，越应该先处理。
- 越靠后：暂时不急，以后再处理。

优先级不是 P1/P2/P3 字段，也不是日期，而是列表中的位置。

这种设计避免了日期系统带来的压力：没有过期任务，没有 Today 负债，也不会因为没按时完成而产生倦怠。

### 6.5 回顾与再整理

每日或每周回顾不按日期催促，只帮助用户整理队列：

- 显示未处理 Memo。
- 显示长期没有打开但仍排在前面的 Memo。
- 显示已完成但未归档的 Memo。
- AI 可以建议“这个 Memo 是否可以拆分、归档或下移”。

回顾是 P1 功能，MVP 先做 Memo 队列和拖拽排序。

## 7. 信息架构

### 7.1 导航结构

手机端底部导航：

- Capture：记录。
- Memos：Memo 队列。
- Search：搜索。

PC 端左侧导航：

- Quick Capture。
- Memo Queue。
- Archived。
- Search。
- Settings。

### 7.2 页面设计

**Capture 页面**

- 顶部或中心大输入框。
- 支持 Enter 换行，Ctrl/Cmd + Enter 保存。
- 保存后生成一张 Memo 卡片。
- AI 解析完成后，Memo 卡片内部显示待办数量。

**Memo Queue 页面**

- 显示一个可拖拽排序的 Memo 卡片列表。
- 每个 Memo 是一个明显的大包围框。
- 卡片内展示原文摘要和前几条内部待办。
- 支持展开/折叠、勾选内部待办、归档 Memo。
- 拖动 Memo 前后顺序表示紧急程度。

**Memo 详情页 / 侧栏**

- 原始内容。
- 创建时间、来源设备。
- 附件。
- AI 摘要。
- Memo 内部待办完整列表。
- 标签。

**Search 页面**

- 搜索 Memo 原文和 Memo 内部待办。
- 支持关键词、标签和状态。
- P1 支持自然语言搜索。

## 8. MVP 功能范围

### 8.1 必做

1. 单用户账号或本地用户模式。
2. 手机/PC 响应式 Web 页面。
3. 快速文本记录。
4. Memo 以大包围框卡片展示。
5. AI 在 Memo 内生成待办项。
6. 用户能编辑、勾选、删除、新增 Memo 内部待办。
7. 用户能拖动 Memo 卡片调整前后顺序。
8. Memo 顺序能跨手机和 PC 同步。
9. Memo 可展开、折叠、完成、归档。
10. 基础搜索。
11. 数据导出为 JSON 或 Markdown。
12. 第一版不出现日期规划、Today、Upcoming、日历、截止提醒或过期任务机制。

### 8.2 P1

1. PWA 安装和离线缓存。
2. 手机语音输入和转文字。
3. 手机分享入口。
4. PC 全局快捷键或托盘快捷输入。
5. 标签系统。
6. Memo 完成进度。
7. 拖拽排序冲突提示和版本历史。
8. AI 队列整理建议。
9. 批量归档。

### 8.3 P2

1. 浏览器扩展。
2. 邮件转发入口。
3. 微信/Telegram/Discord Bot 输入。
4. 图片 OCR。
5. 自然语言搜索。
6. AI 周报，但不自动生成日期计划。
7. 月份地图日期板块：完整月视图，每个日期显示关联待办。
8. MemoTodo 日期接口：在 Memo 内部待办上手动设置所属日期。
9. 自定义 AI 提取规则。
10. 本地优先完整同步。
11. 端到端加密。
12. 开放 API。

## 9. 数据模型草案

### 9.1 User

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 用户 ID |
| name | string | 显示名 |
| created_at | datetime | 创建时间 |
| settings | json | 偏好设置 |

MVP 可以只有一个默认用户。

### 9.2 Memo

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | Memo ID |
| content | text | 原始内容 |
| source | enum | web / mobile / share / voice / api |
| status | enum | active / done / note_only / archived |
| summary | text | AI 摘要 |
| tags | string[] | 标签 |
| sort_order | number | Memo 队列排序，越小越靠前 |
| created_at | datetime | 创建时间，仅用于记录 |
| updated_at | datetime | 更新时间，仅用于同步和历史 |
| device_id | string | 来源设备 |
| sync_version | integer | 同步版本 |
| deleted_at | datetime? | 软删除 |

### 9.3 MemoTodo

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 待办 ID |
| memo_id | string | 所属 Memo |
| title | string | 待办标题 |
| notes | text | 相关上下文 |
| status | enum | todo / done / ignored |
| generated_by_ai | boolean | 是否由 AI 生成 |
| confidence | number? | 0-1 置信度 |
| sort_order | integer | Memo 内部排序 |
| created_at | datetime | 创建时间 |
| completed_at | datetime? | 完成时间，仅用于记录 |

### 9.4 List / Tag

MVP 可以不做复杂清单。标签只作为轻量分类：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 标签 ID |
| name | string | 名称 |
| color | string? | 颜色 |
| sort_order | integer | 排序 |

## 10. AI 设计细节

### 10.1 提取原则

AI 输出必须保守：

- 只提取明确可执行的动作。
- 不把情绪、观点、素材强行变成待办。
- 不建议日期、截止时间或提醒时间。
- 保留原文引用，方便用户检查。
- 所有待办保留在来源 Memo 内部。

### 10.2 Prompt 目标

AI 应返回结构化 JSON：

```json
{
  "summary": "这条 Memo 主要在说什么",
  "memo_todos": [
    {
      "title": "可执行动作",
      "notes": "相关上下文",
      "confidence": 0.82,
      "reason": "原文中出现了明确行动"
    }
  ],
  "note_only_reason": "如果没有待办，说明原因"
}
```

### 10.3 失败处理

- AI 解析失败：Memo 保持 active，提示用户稍后重试。
- AI 返回空待办：标记为 note_only 建议，但用户仍可手动新增待办。
- AI 返回过多待办：默认只展示高置信度，其他折叠。
- 网络失败：先保存 Memo，稍后后台解析。

## 11. 跨端同步设计

### 11.1 MVP 同步策略

MVP 采用“云端权威 + 客户端缓存”：

- 每次记录先写本地缓存，再提交云端。
- 云端返回成功后标记 synced。
- 客户端启动时拉取增量变更。
- 以 `updated_at` 和 `sync_version` 判断新旧。
- 删除采用软删除，避免误删无法恢复。

### 11.2 排序同步策略

Memo 队列顺序是核心数据：

- 每个 Memo 保存 `sort_order`。
- 拖动 Memo 后更新受影响 Memo 的排序值。
- 手机和 PC 都以同一组 `sort_order` 展示。
- 多端同时拖动时，MVP 采用最后一次拖动为准。
- P1 再增加冲突提示和历史恢复。

### 11.3 冲突策略

MVP：

- 不同设备编辑同一 Memo：保留最新版本，并保存旧版本为 history。
- 不同设备编辑同一 MemoTodo：字段级最后写入优先。
- 一端完成待办、另一端编辑待办：完成状态保留，编辑内容合并。

P1：

- 显示冲突提示。
- 支持恢复旧版本。
- 展示同步日志。

### 11.4 离线策略

MVP：

- 离线时允许创建 Memo。
- 离线时允许勾选 Memo 内部待办。
- 离线时允许拖动 Memo，恢复网络后同步排序。
- AI 解析需要联网，离线记录排队。

P1：

- 离线可编辑 Memo 和 MemoTodo。
- 本地全文搜索。

## 12. 日期与提醒策略

首版明确不做日期和提醒系统。

不做：

- 截止日期。
- Today / Upcoming。
- 日历视图。
- 到期提醒。
- 过期任务。
- 按日期滚动任务。

保留的时间字段只用于系统记录：创建时间、更新时间、完成时间。它们用于搜索、同步和历史，不用于给用户制造计划压力。

后续版本加入日期板块，但它不是第一版目标。日期板块采用完整月份地图：展示一个月的所有日期，每个日期格子里显示关联的 MemoTodo。日期关联入口放在 Memo 内部待办项上，由用户手动设置，不由 AI 自动安排。

## 13. 隐私与数据控制

自用工具仍应把数据控制做好：

- 提供完整导出。
- 支持软删除和恢复窗口。
- AI 解析前提示哪些内容会发送到模型服务。
- 设置里允许关闭 AI 自动解析，改为手动解析。
- 设置里允许手动配置 AI API 接口、模型、密钥和发送给 AI 的提示词；系统提供默认提示词。
- P2 再考虑端到端加密；MVP 至少保证传输加密和数据库访问控制。

## 14. 设计原则

### 14.1 产品原则

- **捕获优先**：任何时候进入应用，第一操作都应该是记录。
- **确认优先于自动化**：AI 可以建议，不能污染任务列表。
- **Memo 是容器**：首版围绕 Memo 卡片组织行动，而不是独立任务列表。
- **顺序就是优先级**：拖动 Memo 的前后顺序表达紧急程度。
- **原文永不丢失**：Memo 原文永远保留，内部待办可以被编辑。
- **默认自用**：不为了未来商业化增加复杂度。

### 14.2 UI 原则

- 手机端首屏大输入框 + Memo 卡片队列。
- PC 端左侧快速输入 + 中间 Memo 队列 + 右侧 Memo 详情。
- 每个 Memo 是清晰的大包围框，内部展示 AI 待办。
- 拖拽手柄要明显，排序反馈要稳定。
- 完成 Memo 给轻量反馈，不做过度游戏化。

## 15. 验收标准

MVP 完成后应满足：

1. 在手机浏览器和 PC 浏览器都能登录并看到同一批 Memo。
2. 用户能在 3 秒内完成一条 Memo 记录。
3. 保存 Memo 后，即使 AI 失败，原始内容也不会丢失。
4. AI 能从一段自然语言中提取 0-N 个 Memo 内部待办。
5. AI 待办不会脱离 Memo 成为外部任务。
6. 用户能编辑、勾选、删除、新增 Memo 内部待办。
7. 用户能拖动 Memo 调整前后顺序。
8. 手机和 PC 的 Memo 顺序保持一致。
9. Memo 能完成、归档、恢复查看。
10. 搜索能同时找到 Memo 原文和内部待办。
11. 数据可以导出为 JSON 或 Markdown。
12. 应用内不存在付费、会员、订阅或升级入口。
13. 第一版应用内不存在 Today、Upcoming、截止日期、到期提醒或过期任务机制。

## 16. 建议开发顺序

第一阶段：基础数据闭环

1. 建立 Memo / MemoTodo 数据模型。
2. 实现响应式 Web 基础布局。
3. 实现快速记录。
4. 实现 Memo 卡片队列。
5. 实现 Memo 内部手动新增待办。

第二阶段：AI 转化闭环

1. 接入 AI 解析服务。
2. 实现 AI 生成 Memo 内部待办。
3. 实现编辑、勾选、删除内部待办。
4. 实现 Memo 展开、折叠、完成和归档。

第三阶段：跨端可用

1. 实现登录或单用户鉴权。
2. 实现云端同步。
3. 同步 Memo 内容、内部待办和 Memo 排序。
4. 实现基础搜索。
5. 实现导出。

第四阶段：体验增强

1. PWA 安装。
2. 离线创建 Memo。
3. 离线拖动排序。
4. 语音输入。
5. 队列整理回顾。

## 17. 主要风险

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| AI 提取不准 | 误把背景信息变成内部待办，或漏掉关键行动 | AI 待办保留在 Memo 内部，用户可编辑、删除、忽略 |
| 工具太复杂 | 做成小 Notion 或小滴答 | MVP 只保留 Capture、Memo Queue、Search |
| 同步不可靠 | 手机记录 PC 看不到，或排序不一致 | 先做简单云端权威同步，明确同步 Memo 顺序 |
| 记录堆积不处理 | Memo 越积越多 | 通过拖动排序、完成和归档维持队列 |
| 自用动力不足 | 开发完不用 | 首版先服务你最真实的 3 个场景，而不是泛化 |

## 18. 下一步建议

建议下一步不要立刻开写全部功能，而是先确认三个自用场景：

1. Memo 卡片内部默认展示几条待办最舒服：3 条、5 条，还是全部展开？
2. Memo 完成后是自动归档，还是保留在队列里由你手动归档？
3. 多端排序冲突时，是简单采用最后一次拖动，还是需要提示你确认？

在没有更多偏好的情况下，我建议首版按这个最窄目标开工：

> 一个 PWA：打开就能写 Memo；AI 在 Memo 包围框内部生成待办；你拖动 Memo 排紧急程度；手机和 PC 的 Memo 顺序同步。

这个版本足够小，但已经验证了 MemoTask 与普通待办、普通笔记的核心差异：它用 Memo 容器承载任务，用顺序表达紧急程度，不用日期制造压力。
